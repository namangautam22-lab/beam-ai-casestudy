Frame the Problem

The estimator's real problem is knowing when to stop

Beam AI surfaces hundreds of annotations per drawing. The AI handles detection. But the estimator carries the professional risk. If the exported BoQ is wrong, it's their name on it.

The result: estimators either check everything (slow) or check selectively and absorb personal liability for what they missed. What's missing is a defensible stopping point — a system that says they've covered enough high-risk surface area to sign off.

Solution Architecture

Three-layer architecture

Each layer builds on the last. The key constraint: no new ML models. Everything surfaces what Beam already computes.

🎯
Layer 1: Confidence Engine
Scores each annotation 0–100 based on symbol clarity, bounding box overlap, drawing scale consistency, and historical accuracy for that item type.
→
⚡
Layer 2: Risk Prioritizer
Combines confidence score, estimated dollar value, and item rarity into a ranked queue. Low confidence + high value = top of queue.
→
✓
Layer 3: Quick Review UX
Keyboard-first interface surfaces drawing context inline. Progress bar and export gating prevent premature BoQ submission.
Key Trade-off

False positives are acceptable. Cost of unnecessary check: ~10 seconds. Cost of missed AHU: $40,000+. System biases toward over-flagging. If estimators show frustration with false positives, lower the HIGH threshold from 75% to 65% confidence and monitor acceptance rate.

Without field integration, confidence scores remain static, manual correction logs in Phase 2 serve as the interim calibration mechanism.

Assumptions: stated explicitly

  1. Confidence scores already exist in Beam's ML pipeline. This is a UI surface change, not a new model.
  2. Dollar values come from RSMeans 2024 HVAC reference data, not live quote feeds.
  3. "Review complete" ≠ 100% coverage. Threshold: all High-Risk items cleared + 60% overall.
  4. Confidence scores are static until field integration exists. Manual correction logs in Phase 2 act as a proxy signal. [ASSUMPTION: explicitly unvalidated until Phase 1 data exists.]

Rollout Plan

Three-phase strategy

Each phase ships something estimators can use immediately, then instruments the data needed to justify the next phase.

Phase 1Weeks 1–3
Signal Only
Add confidence badge to existing annotation list. No reordering. Feature flag to 2 beta estimators.
Exit criterion
Estimators describe unprompted which items are high vs low confidence. Validated by session recording.
Phase 2Weeks 4–10
Ranked Queue + Review Tracking
Risk-ranked list, keyboard shortcuts, progress bar, inline detail view. Export ungated. Manual correction log introduced as interim calibration signal.
Exit criterion
Median review time under 40 min across 10 sessions. High-risk items opened before medium items in 80%+ of sessions.
Phase 3Weeks 11–18
Export Gating + Session Summary
Block export until all HIGH items actioned. Session summary shown before export. Flagged items routed to project lead. Post-export error tracking via manual logs; Procore integration if available.
Exit criterion
Post-export error rate under 1.5% across 20+ export events.
Phase 3 gating dependency

Phase 3 gating only ships if Phase 2 shows estimators trust the confidence ranking. If trust signals are weak, Phase 3 scope reduces to session summary only — no hard gate.

Risk Reflection

The one question that determines whether this works

Are confidence scores actually calibrated?

Before building the ranked queue, audit 50 historical drawings. If 61% confidence and 85% confidence items have the same real-world correction rate, the prioritization is noise and the product premise fails.

Test: do low-confidence items get corrected at 2x the rate of high-confidence items? If yes — build. If no — pivot to dollar-value-only prioritization, no confidence signal needed.

Secondary risk: export gating in Phase 3 creates friction if confidence scores aren't trusted yet. Mitigation: Phase 2 keeps export ungated — observe behaviour before enforcing. Trust must be earned before gates are added.
